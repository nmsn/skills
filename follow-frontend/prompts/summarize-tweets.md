# 如何 Remix 推文内容

## 目标

把一位 builder 的推文转化成**一段有价值的工程师情报**，而不是翻译推文。

## 处理规则

### 筛选

只保留以下类型的内容：
- 技术观点 / 架构决策（"我们为什么选择 X 而不是 Y"）
- 新版本 / 新功能发布（附链接）
- 正在解决的技术问题（某个 bug、某个设计权衡）
- 对某个技术趋势的判断（Bun vs Node、RSC 争议等）
- 真实的工程数据（性能数字、bundle size、冷启动时间）

直接跳过：
- 纯转推、点赞、问候
- 活动宣传、演讲预告（除非内容本身有干货）
- 与前端/Node/Bun/CSS/构建工具无关的内容

### 写法

**格式：**
```
[名字]（[身份简介]）[一句核心观点或动态]：[1-3 句展开]
🔗 [原文链接]
```

**示例（好）：**
```
Evan You（Vue / Vite 作者）分享了 Vite 6 的 Environment API 设计思路：把 client、SSR、edge worker 的构建上下文彻底解耦，每个 environment 有独立的模块图。这解决了长期以来 SSR 插件无法复用 client 配置的问题。
🔗 https://x.com/youyuxi/status/xxx
```

**示例（差，避免）：**
```
尤雨溪发了一条推文说 Vite 6 出来了。
🔗 https://x.com/...
```

### 合并

如果同一个人发了多条关于同一话题的推文，合并成一段，只保留最有价值的那条链接。

### 长度

每位 builder：**2-4 句话**，超过 4 句说明你没有提炼。
